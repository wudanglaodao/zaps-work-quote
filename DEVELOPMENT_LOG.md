## v0.1.0 — 从成本计算到专业报价

发布日期：2026-07-11
版本状态：已发布

这是 zaps.work 的第一个公开版本。

我们从一个很具体的问题开始：小型经营者和创客知道材料用了多少、机器运行了多久，却不一定能快速算清一笔订单的真实成本，更难判断报价能否达到目标利润。

首个上线工具是 **3D 打印成本计算器**。它把材料、打印时间、设备费率、人工、损耗和失败风险放进同一个透明的计算过程，再根据目标毛利生成建议报价。用户可以添加多个项目，并把结果导出为 PDF 报价单或 CSV 文件。

### 本版新增

- 上线 3D 打印成本计算器，支持多项目成本与报价计算。
- 支持材料、设备、人工、损耗、失败风险和其他成本的独立配置。
- 根据目标毛利计算建议报价，并展示成本构成。
- 支持 PDF 报价单与 CSV 数据导出。
- 提供英语、繁体中文、德语、日语、西班牙语、法语、巴西葡萄牙语和韩语版本。
- 加入可搜索的语言与货币选择器，并根据语言地区提供合理的货币默认值。
- 完成响应式布局，让手机用户可以先看到计算表单和关键结果。
- 上线隐私说明、站点分析、健康检查、站点地图、`robots.txt` 与 `llms.txt`。
- 开放源代码与 GitHub 反馈入口，计算逻辑和数据边界可以被公开检查。

### 我们特别在意的事

**计算必须透明。** zaps.work 不用一个看不见的“市场均价”替用户做决定。每一项成本、费率和利润目标都可以查看和调整。

**客户资料默认留在浏览器。** 姓名、邮箱、电话、地址、公司信息、报价编号和备注等敏感内容不会进入产品分析数据库。我们只记录经过白名单限制的匿名使用数据，用来判断工具是否真正有帮助。

**工具应该先能用，再讲故事。** 在手机和桌面端，计算器都尽量出现在页面最前面。我们也修复了数字输入、步进按钮、移动端排列和长列表滚动等首版体验问题。

### 幕后建设

- 使用 Next.js、React 与 TypeScript 建立多语言工具平台。
- 将计算逻辑拆分为可测试的纯函数，并加入单元测试与发布质量检查。
- 使用 Vercel 承载预览与生产环境，使用 Supabase 保存隐私安全的匿名分析事件。
- 建立 canonical、`hreflang`、多语言路由和搜索引擎索引基础。

### 下一步

接下来我们会继续打磨 3D 打印从“成本输入”到“客户报价”的完整流程，并验证更多真实工作场景。正在研究和开发的方向包括压力清洗、清洁服务、激光切割等，但它们会在计算模型、报价内容和实际用户验证完成后再公开发布。

如果你正在用 zaps.work 处理真实报价，欢迎通过 [GitHub](https://github.com/wudanglaodao/zaps-work-quote) 提交反馈。

## v0.2.0 — 激光切割报价垂直切片

发布日期：2026-07-11
版本状态：开发完成，待发布

第二个制造场景围绕激光切割报价展开。它沿用 zaps.work 的通用报价文档和隐私边界，同时把激光切割真正需要的面积、材料厚度、切割长度、穿孔数量、机器时间、准备和后处理成本放进一个可调整的模型。

### 本版新增

- 上线激光切割成本计算器与报价单页面。
- 支持金属、亚克力、木板等材料分类，以及平方英尺和平方米单位。
- 支持材料损耗、切割时间、穿孔数量、机器费率、准备时间、搬运人工和后处理成本。
- 可选成本区默认关闭，减少首次填写干扰；打开后才展示额外成本字段。
- 沿用 PDF、CSV 和复制摘要导出，并把激光切割的匿名汇总事件纳入分析白名单。
- 加入所有已支持语言的本地化页面、canonical、`hreflang`、FAQ 结构化数据和工具站点地图入口。

### 下一步

下一步继续验证激光切割的真实报价输入，再决定是否扩展到清洁服务或其他本地服务场景。

## 后续版本记录模板

> 发布新版本时复制下面的结构，将新版本放在本文顶部，并删除不适用的小节。

```markdown
## vX.Y.Z — 一句话说明本版价值

发布日期：YYYY-MM-DD
版本状态：已发布

用两三句话说明这个版本解决了什么问题，以及用户为什么会在意。

### 本版新增

- 用户可以感知的新功能或变化。

### 体验改进

- 操作、性能、移动端、可访问性或内容方面的提升。

### 问题修复

- 对用户有影响的重要修复，不需要罗列所有内部提交。

### 幕后建设

- 值得公开说明的架构、测试、隐私、安全或发布能力。

### 下一步

简要说明已经确认的方向，避免承诺未经验证的发布日期。
```

## 记录约定

- 使用语义化版本号：重大不兼容变化为主版本，新功能为次版本，修复为补丁版本。
- 只记录已经进入生产环境的内容；开发中的功能放在“下一步”，不要写成已经发布。
- 从用户价值开始写，内部重构只有在影响可靠性、速度、隐私或后续能力时才记录。
- 发布日期以生产环境上线日期为准。
- 每次上线后补充日志链接、最终版本号和必要截图，即可整理为博客文章。

## 2. 一句话定位

> zaps.work 是面向创客、个体经营者和小微企业的免费成本计算与专业报价工具平台。

对外英文定位：

> Quote tools for real-world work.

核心用户承诺：

> 算清一项工作的真实成本，守住目标利润，并快速生成可以发给客户的清晰报价。

## 3. 产品愿景

让缺少财务、定价和专业软件能力的小型经营者，也能用可靠、透明的方法完成每一次成本核算和客户报价。

zaps.work 的长期目标不是成为一组彼此无关的在线计算器，而是成为适用于多种真实工作场景的轻量报价基础设施：

```text
工作场景
  -> 场景化输入
  -> 成本模型
  -> 定价规则
  -> 利润校验
  -> 客户报价
  -> 导出与复用
```

## 4. 用户与核心问题

### 4.1 核心用户

第一阶段聚焦两类用户：

1. **小型制造与创客经营者**
   - 3D 打印工作室、激光切割店、小型 CNC 加工商和接单型创客。
   - 通常使用经验、简单表格或材料费倍数进行报价。
   - 需要纳入材料、设备时间、人工、损耗、失败风险、最低收费和目标利润。

2. **本地服务经营者**
   - 清洁、压力清洗、维修、安装、园艺、油漆等个人或小团队。
   - 需要把工作范围、工时、人员、耗材、路程、附加服务和最低上门费转化为固定报价。

第二阶段可扩展到自由职业者和创意服务团队，但在制造与本地服务两种核心引擎得到验证之前，不作为主要投入方向。

### 4.2 典型用户特征

- 有真实客户和真实订单，但没有专职财务或报价人员。
- 报价频率不足以支撑复杂、昂贵的行业软件。
- 能提供工作参数，却不一定理解毛利、加价率和成本分摊。
- 希望迅速回复客户，同时避免漏算成本和低价接单。
- 需要输出看起来专业、内容完整的报价文件。
- 对注册、上传客户资料和复杂配置较为敏感。

### 4.3 核心问题

用户真正要完成的任务不是“做一次计算”，而是连续回答三个问题：

1. **这项工作实际会花掉我多少钱？**
2. **为了达到目标利润，我应该报多少钱？**
3. **怎样把结果整理成客户能够理解和接受的报价？**

现有替代方案通常只能解决其中一部分：

- 心算或经验倍数很快，但容易漏算人工、设备、损耗和风险。
- 通用表格灵活，但搭建门槛高、公式容易出错，也不适合直接发给客户。
- 市场均价计算器给出参考区间，却无法反映用户自己的成本结构。
- CRM、ERP 和垂直报价软件功能完整，但对小型经营者通常过重、过贵或需要注册配置。

## 5. 价值主张

zaps.work 提供一条从工作参数到客户报价的完整短路径：

### 5.1 算得真实

- 按具体行业场景设计输入项和计算模型。
- 明确展示材料、设备、人工、损耗、失败风险和其他成本。
- 区分加价率与毛利率，避免常见定价错误。
- 使用用户自己的费率和成本，不把市场均价伪装成正确答案。

### 5.2 讲得明白

- 结果可解释，用户能看到报价如何形成。
- 默认值帮助新用户开始，同时允许有经验的用户调整关键参数。
- 通过提示、校验和警告暴露不合理或高风险输入。

### 5.3 发得出去

- 将计算结果转化为客户可读的正式报价。
- 支持多项目、报价有效期、范围、假设、排除项和条款。
- 支持 PDF、CSV 和文本摘要等轻量导出方式。

### 5.4 用得放心

- 核心工具免费、无需登录即可使用。
- 客户姓名、联系方式、地址、报价编号、项目说明等敏感内容默认只保留在浏览器中。
- 仅采集经过白名单允许的匿名产品使用事件。

## 6. 市场位置与差异化

zaps.work 位于“单一在线计算器”和“完整经营管理软件”之间。

| 方案 | 优点 | 主要不足 | zaps.work 的选择 |
| --- | --- | --- | --- |
| 心算、经验倍数 | 快、无学习成本 | 容易漏项，利润不可控 | 保留速度，同时提供完整成本结构 |
| 通用表格 | 灵活、可自定义 | 搭建和维护成本高 | 提供开箱即用的场景模型 |
| 市场均价计算器 | 简单、适合获客 | 不能反映个人成本 | 使用用户自己的成本和目标利润 |
| 报价模板 | 输出专业 | 不负责成本与定价 | 计算和报价在同一流程完成 |
| CRM、ERP、垂直 SaaS | 功能全面 | 复杂、昂贵、注册门槛高 | 无登录、轻量、任务优先 |

核心差异化不是“免费计算”，而是以下组合：

> 场景化成本模型 + 可解释定价 + 客户级报价输出 + 无登录隐私保护。

## 7. 产品原则

### 7.1 工具优先

用户进入页面后应尽快开始计算。内容、SEO 和商业化模块不能阻挡主要任务。

### 7.2 场景具体，共享底层

不同工作使用不同的输入和公式，不强行生成一张万能表单；页面框架、定价规则、结果结构、报价文档、导出和分析能力尽量复用。

### 7.3 透明胜过黑盒

所有关键成本和计算逻辑都应可查看、可调整。产品提供建议，不冒充当地市场价格或替用户作出最终经营决策。

### 7.4 渐进复杂

首次使用保持简单；专业费率、风险参数和高级设置按需展开。默认值必须有依据且可编辑。

### 7.5 专业输出

报价结果不仅要正确，还要能够直接进入真实客户沟通。范围、假设、排除项和有效期与总价同样重要。

### 7.6 隐私默认

能在浏览器本地处理的数据不上传。任何账户、云端历史记录或文件解析能力，都必须先明确同意、保存期限和删除机制。

### 7.7 先验证，再自动化

先用透明的手工输入验证需求和模型，再考虑 DXF、G-code、CAD 等文件解析或自动报价能力。

## 8. 平台结构

### 8.1 统一品牌

- `zaps.work` 是长期主品牌和统一域名。
- 所有工具使用路径式 URL，积累同一域名的搜索权重和用户认知。
- 页面按工具、模板和指南组织，不为单个场景拆分独立子品牌。

### 8.2 产品模块

```text
zaps.work
  ├── Tools：场景化成本计算与报价工具
  ├── Templates：可复用报价模板
  ├── Guides：成本、定价、毛利和行业报价指南
  └── Account（待验证）：保存、复制和管理历史报价
```

### 8.3 场景扩展标准

只有同时满足以下条件的场景才进入建设：

1. 用户存在明确、重复发生的报价任务。
2. 关键输入能够由用户可靠获得。
3. 成本模型可以解释，且不依赖无法验证的黑盒估算。
4. 能复用共享报价引擎和导出能力。
5. 存在清晰的搜索需求、用户渠道或访谈证据。
6. 误差和责任边界能够通过假设、提示或人工复核说明。

### 8.4 新工具标准交付流程

每个新场景都必须作为一个完整的“工具 + 报价 + 内容 + 搜索入口”垂直切片交付，不能只上线计算表单。标准流程如下：

```text
场景与关键词验证
  -> 透明计算模型与单元测试
  -> 场景化计算器或报价工具
  -> 客户报价预览与 PDF/CSV/摘要导出
  -> 多语言页面与页面级 SEO
  -> 至少 1 篇对应的操作或定价指南
  -> 工具、指南和相关场景之间的内链
  -> Sitemap 与 llms.txt 更新
  -> 生产验证与搜索引擎提交
  -> 使用、导出和自然搜索数据复盘
```

每次发布新工具时，以下项目属于发布硬门槛：

1. 完成目标关键词、搜索意图和页面标题说明，不用与场景不匹配的通用文案。
2. 提供完整的 canonical、`hreflang`、Open Graph、Breadcrumb、WebApplication 和 FAQ 结构化数据。
3. 将所有可索引语言的工具 URL 加入工具 Sitemap；新增指南时同步加入指南 Sitemap。
4. 更新 `public/llms.txt`，加入新工具、指南、能力说明和正确的公开 URL。
5. 至少发布 1 篇与工具直接对应的操作指南、定价指南或完整报价教程，并从工具页双向内链。
6. 检查 8 个现有语言版本的 UI、元数据、数字格式、货币格式和 PDF 输出；未完成的语言不得发布薄翻译页面。
7. 验证计算逻辑、分析事件、移动端、导出、Sitemap 和生产 URL 后，才将状态改为“已上线”。

## 10. 产品路线图

路线图采用“阶段目标 + 验证门槛”，不以未经评估的固定日期替代产品验证。

### 阶段 1：稳定现有三个工具与增长基线

目标：证明用户愿意从成本输入走到报价导出，并明确最先起量的工具、关键词和语言市场。

重点工作：

- 监测 3D 打印、压力清洗与激光切割的访问、有效计算和报价导出漏斗。
- 补齐激光切割操作与定价指南，并与工具页建立双向内链。
- 根据 Search Console 数据持续优化页面标题、说明、FAQ 和正文。
- 验证工具 Sitemap、指南 Sitemap、canonical、`hreflang` 和 `llms.txt` 与线上状态一致。
- 建立隐私安全的计算完成与导出漏斗。
- 收集真实用户对默认值、缺失成本、公式和报价格式的反馈。

进入下一阶段的门槛：

- 核心公式和导出流程稳定，无高优先级准确性问题。
- 能持续测量访问、有效计算、报价导出和回访。
- 获得真实接单用户对计算项与报价结果的定性验证。

### 阶段 2：发布清洁服务报价生成器（开发完成，待发布）

目标：用第二个本地服务场景验证压力清洗建立的服务报价引擎能否低成本复用。

重点工作：

- 支持住宅、办公室、搬家与深度清洁等场景。
- 支持房间、卫生间、面积、频率、工时、耗材、交通、最低收费和附加服务。
- 复用现有毛利、税费、PDF、CSV、摘要与隐私安全分析能力。
- 同步完成 8 种语言页面、页面级 SEO、工具 Sitemap、`llms.txt` 和至少 1 篇清洁报价操作指南。
- 用真实报价验证默认工时、附加服务和最低收费模型。

### 阶段 3：发布房屋油漆报价计算器

目标：继续扩展高意图本地服务场景，并验证面积、材料覆盖率、遍数与准备工时模型。

- 支持墙面、天花板、门窗面积，室内与室外项目。
- 支持涂刷遍数、油漆覆盖率、材料、表面修补、遮蔽、人工、设备和交通。
- 支持最低收费、目标毛利、税费和客户报价导出。
- 同步完成 8 种语言页面、页面级 SEO、Sitemap、`llms.txt` 和至少 1 篇油漆报价操作指南。

### 阶段 4：发布手工参数版 CNC 加工成本计算器

目标：在不依赖文件解析的情况下，扩展更复杂的制造报价场景。

重点工作：

- 支持材料与毛坯、机器时间、设置、编程、刀具损耗、人工、后处理和废品风险。
- 支持数量、最低收费、目标毛利与报价导出。
- 在真实手工输入得到验证前，不投入 CAD、CAM 或 G-code 自动解析。
- 同步完成 8 种语言页面、页面级 SEO、Sitemap、`llms.txt` 和至少 1 篇 CNC 成本操作指南。

### 阶段 5：根据数据选择下一批场景

目标：根据自然搜索、有效计算、报价导出、回访和开发复用成本，选择园艺／草坪服务、汽车美容或其他明确场景，不按固定清单盲目铺量。

### 暂缓方向

- CNC 基于 CAD 的全自动即时定价。
- 报价单转换为发票、收据或付款管理的单据工作流。
- 账户、历史单据、客户管理和团队协作。
- 未经需求验证的大规模场景铺设。
- 为增加功能数量而制作的通用万能计算器。
- 强制登录、强制上传客户资料或强制云端保存。
- 在流量尚未形成前优先投入广告系统。
- 在缺少真实订单校准时承诺“准确市场价格”。

## 11. 成功指标

### 11.1 北极星行为

**有效报价导出数**：用户完成有效计算后，导出 PDF、CSV 或复制客户摘要。

这一行为同时反映计算结果是否可信、报价输出是否有用，以及产品是否真正完成了用户任务。

### 11.2 核心漏斗

```text
工具页访问
  -> 修改有效输入
  -> 生成有效结果
  -> 查看报价预览
  -> 导出或复制报价
  -> 再次使用
```

建议跟踪：

- 工具页到有效计算的转化率。
- 有效计算到报价导出的转化率。
- 各导出方式的使用比例。
- 7 日和 30 日匿名回访率。
- 每次会话的项目数量与场景使用深度。
- 公式警告、校验失败和导出失败率。
- 自然搜索进入工具后的有效计算率。
- 用户反馈中“漏项”“难懂”“不可信”和“无法发送”的问题数量。

### 11.3 数据边界

不得为了衡量产品而收集客户姓名、联系方式、地址、公司资料、报价编号、项目名称、图纸版本、备注、条款或完整报价快照。

具体数值目标应在生产分析稳定、取得首个完整基线周期后设定。没有基线前，不使用缺乏依据的增长百分比作为承诺。

## 12. 关键假设与验证计划

| 假设 | 验证方式 | 通过信号 |
| --- | --- | --- |
| 用户需要的不只是成本结果，而是可发送报价 | 分析计算到导出漏斗；用户访谈 | 有稳定比例的有效计算进入导出 |
| 透明公式比黑盒市场均价更可信 | 可用性测试；反馈分类 | 用户能解释报价构成并愿意调整参数 |
| 无登录是首次使用的重要优势 | 首次使用测试；登录意愿调查 | 无登录完成率高，账户需求主要来自复用场景 |
| 制造与本地服务可以共享报价底层 | 发布第二类场景并比较开发与使用结果 | 共享结果、文档和导出能力无需大量特例 |
| 搜索可以成为主要获客渠道 | 工具页和指南页自然流量分析 | 高意图页面带来有效计算而非只有浏览 |
| 高频用户愿意为保存和品牌化付费 | 原型测试；意愿访谈 | 用户主动要求复用资料、历史报价或品牌模板 |

## 13. 主要风险与应对

### 13.1 场景过多导致产品失焦

应对：所有新场景必须通过扩展标准和验证门槛；优先做完整垂直切片，不追求工具数量。

### 13.2 结果被误认为市场标准价

应对：明确结果来自用户输入和目标利润；展示公式、假设和可编辑参数；不承诺当地成交价。

### 13.3 默认值制造虚假准确感

应对：标注默认值含义和适用范围，鼓励替换为自身数据，对高影响参数提供警告。

### 13.4 免费流量无法形成可持续业务

应对：先验证使用、回访和搜索增长，再测试非干扰广告及高频用户付费能力，不提前牺牲核心体验。

### 13.5 客户数据带来隐私与合规风险

应对：坚持本地处理和分析白名单；未来云端保存必须具备明确同意、访问控制、导出和删除机制。

### 13.6 高复杂制造场景损害可信度

应对：将产品定位为透明报价工作表和辅助判断；复杂场景保留人工复核，不在缺少历史数据与专家校准时自动承诺价格。

## 14. 产品决策检查清单

每项新功能或新场景立项前回答：

1. 它服务的是哪类核心用户和哪项真实报价任务？
2. 它减少了漏算成本、错误定价还是报价制作时间？
3. 用户能否理解并校验结果？
4. 是否能够复用现有成本、定价、报价和导出能力？
5. 是否引入客户敏感数据，能否在浏览器本地完成？
6. 有什么使用、搜索、访谈或订单证据支持优先级？
7. 成功和停止投入的判断标准是什么？
8. 如果不做，是否会影响当前核心任务的完整闭环？

如果无法清晰回答以上问题，该需求不应直接进入开发。

## 15. 未来 90 天建议重点

1. 第 1–2 周：稳定现有三个工具的数据漏斗，补齐激光切割指南，并核对全部 Sitemap、`llms.txt` 与多语言 SEO 输出。
2. 第 3–5 周：发布清洁服务报价生成器，同时发布至少 1 篇对应操作指南和完整搜索入口。
3. 第 6–8 周：发布房屋油漆报价计算器，同时发布至少 1 篇对应操作指南和完整搜索入口。
4. 第 9–12 周：开发手工参数版 CNC 加工成本计算器，同时准备对应操作指南和搜索入口。
5. 每个新工具必须同步完成页面级 SEO、8 种语言、结构化数据、内链、工具 Sitemap、指南 Sitemap 和 `llms.txt` 更新，不把这些工作留到上线之后。
6. 每月复盘自然流量、搜索查询、有效计算、报价导出、回访、用户反馈和每个场景的开发复用成本，再决定下一项场景。

## 16. 结论

zaps.work 的机会不在于做更多简单计算器，而在于把小型经营者最容易出错的“成本—定价—报价”过程变成一条透明、专业且无负担的工作流。

近期应专注证明两件事：

1. 用户会把计算结果转化为真实客户报价。
2. 同一套底层能力可以跨越制造与本地服务两类场景。

在这两点得到验证后，再扩展场景、账户、自动化和商业化，产品才能从单点工具成长为可持续的报价平台。
